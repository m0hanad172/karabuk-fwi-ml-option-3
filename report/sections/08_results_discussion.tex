\chapter{Results and Discussion}

\section{Stage 1 Regression Results}

The Stage 1 model was evaluated as a continuous Fire Weather Index (FWI)
regression model. Its task is to estimate daily FWI from the locked set of 34
engineered input features described in Chapter 3. The deployed model is a
\texttt{HistGradientBoostingRegressor}; its output,
\texttt{predicted\_fwi}, is the numerical risk estimate shown in the dashboard
and passed to the high-risk decision layer.

Table~\ref{tab:stage1-regression-metrics} reports the verified Stage 1
metrics from the repository metadata. The out-of-fold validation result was
RMSE \(=4.111\), MAE \(=2.918\), and \(R^2=0.835\). The 2025 holdout result was
RMSE \(=7.259\), MAE \(=5.098\), and \(R^2=0.819\). The increase in error on
the later holdout period is expected because it tests the model on a
chronologically later season, but the holdout \(R^2\) still indicates that the
model captures much of the variation in the FWI target.

\section{Stage 2 High-Risk Classification Results}

Stage 2 was evaluated as a high-risk classification layer. It uses a
\texttt{RandomForestClassifier} with four inputs:
\texttt{predicted\_fwi}, \texttt{rh}, \texttt{ws}, and
\texttt{fuel\_drying\_rate}. The class target is \texttt{target\_ge\_35},
where a day is high risk when \( \mathrm{FWI} \geq 35 \). The final stored
probability threshold is 0.10.

Table~\ref{tab:stage2-classification-metrics} gives the final classifier
values verified from the repository metadata. At threshold 0.10, the Stage 2
classifier achieved precision \(=0.849\), recall \(=0.900\), F1-score
\(=0.874\), and accuracy \(=0.929\), with 45 true positives, 8 false
positives, 5 false negatives, and 126 true negatives.

\section{YOLOv8 Fire and Smoke Detection Results}

The visual monitoring stage uses the active YOLOv8 \texttt{best.pt} detector
for two object classes: fire and smoke. The detector provides visual evidence
through bounding boxes, class labels, and confidence values. This is different
from the prediction model: prediction estimates fire-weather risk, while YOLOv8
detection identifies visible evidence in camera or drone-ready frames.

Object detection should be evaluated with object-detection metrics rather than
generic image-classification accuracy. Precision measures how many detections
are correct, recall measures how many relevant objects are found, and
mAP@0.5 / mAP@0.5:0.95 summarize localisation and classification quality
across confidence and overlap thresholds. Table~\ref{tab:yolo-detection-metrics}
therefore reports precision, recall, and mAP values from the uploaded YOLOv8
training/evaluation output.

\section{Dashboard and System Validation Results}

The dashboard demonstrates how the FireWatch workflow can be used as an
operator-facing decision-support interface. It brings together risk estimates,
monitoring feeds, detection alerts, system/model status, and map context
instead of leaving each component as a separate tool.

The main validation result from the software layer is integration. Backend
tests verify API behaviour, inference, decision logic, scheduler behaviour,
monitoring endpoints, detection alert persistence, read/unread state, soft
delete behaviour, and separation between detection alerts and prediction
history. A local frontend lint and production build check also confirmed that
the React/TypeScript dashboard compiles successfully.

\section{Limitations}

The Stage 1 regression model still has prediction error. Although the
out-of-fold and holdout results show useful performance, unusual weather
patterns and difficult high-FWI days can still produce larger errors. The
predicted FWI value should therefore be interpreted as a decision-support
estimate, not as a replacement for official fire-danger products.

The Stage 2 classifier improves high-risk decision support but does not remove
false positives or false negatives. False positives can increase monitoring
load, while false negatives are more serious because they may reduce attention
during dangerous conditions. This is why FireWatch shows the probability and
decision reason rather than hiding the result behind a single unexplained
label.

The YOLOv8 detection stage is also limited. The uploaded training/evaluation
curves provide useful object-detection evidence, but they are not a substitute
for extended field validation in Karabük forests, under drone motion, varied
lighting, glare, fog, and different smoke densities. Multi-frame confirmation,
glare filtering, bounding-box area filtering, and larger local evaluation
sets remain future work.

The persistence and drone-ready layers are intentionally prototype-oriented.
SQLite is appropriate for a local single-machine demonstration, but a deployed
multi-user system would require a server database, access control, backups,
and operational monitoring. The drone layer supports mock and Tello-oriented
workflows with operator safety gates; it does not claim certified autonomous
drone patrol.

\section{Discussion}

The results answer the research question by showing that FireWatch can combine
daily wildfire risk prediction, visual fire/smoke detection, persistent alert
management, and dashboard-based decision support for the Karabük region. Each
module contributes a different type of evidence. Stage 1 provides continuous
fire-weather risk estimation, Stage 2 supports high-risk classification,
YOLOv8 provides visual evidence, SQLite preserves operational history, and the
dashboard makes the workflow reviewable.

The two-stage prediction design is useful because it separates risk intensity
from high-risk decision support. A continuous FWI estimate helps operators
understand severity, while the Stage 2 classifier focuses on reducing missed
high-risk days near the decision boundary. This is more informative than a
single binary model because it preserves both the estimated risk level and the
operational decision.

The detection model adds a complementary role. Weather-based prediction cannot
confirm that fire or smoke is already visible, and camera detection alone does
not explain whether the wider weather conditions are dangerous. By keeping
prediction and detection separate but visible in the same dashboard, FireWatch
supports both preparedness and visual evidence review.

Overall, FireWatch is strongest as a regional academic decision-support
prototype. It does not claim official emergency integration, field
certification, or production autonomous drone operation. However, the results
demonstrate a complete engineering workflow that connects machine learning,
computer vision, persistence, scheduling, and dashboard monitoring into a
coherent FireWatch prototype.
